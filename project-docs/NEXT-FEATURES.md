# Rental Buddy 下一階段開發文件

最後更新：2026-05-09（F-007 附件上傳 MVP S5／完成）

## 目標

這份文件用來管理「接下來要加的功能」，讓需求、切片、驗收標準與發布紀錄集中在同一處。

---

## 開發原則

1. 先定義使用情境，再定義 UI 與資料欄位。
2. 每個功能都要有「可驗收標準」。
3. 每次實作盡量小步快跑（1 個功能 = 1 個可演示版本）。
4. 若涉及 GitHub Pages，推送前需同步 `docs/`。
5. 若功能牽涉第三方服務（地圖、登入、金流），先做最小可行串接（MVP）再擴充。

## 功能細節草案（待逐項討論定稿）

### F-001 地圖選址（地址定位）
- **MVP 目標**：從查核表開啟內嵌地圖，直接點選位置並回填相似地址。
- **細節草案**：
  - 在查核表基本資料區新增「在地圖上選地址」按鈕。
  - 使用 OpenStreetMap + Leaflet，不需要 Google API key。
  - 點地圖後儲存座標，並用免費反查服務回填相似地址。
  - 反查地址可不精準，仍允許使用者手動修正地址文字。
- **本輪定稿**：
  - 入口位置：查核表地址欄；報告頁提供返回查核表選址入口。
  - 資料規則：保存 `latitude` / `longitude` 與可手動修正地址。
  - 範圍策略：先做免費 MVP，之後再評估更精準的商用地理服務。

### F-002 查核表版面重構（手機優先）
- **MVP 目標**：在看房現場更快勾選，不需要頻繁捲動或找按鈕。
- **細節草案**：
  - 上方摘要列固定：目前紀錄、地址、月租金、完成度。
  - 分類切換改成下拉選單（省空間，提升同屏資訊量）。
  - 底部動作列固定：立即儲存 / 查看報告 / 新增紀錄。
- **本輪定稿**：
  - 版面優先：同屏資訊量優先（`more_info_visible`）。
  - 導覽方式：分類下拉（`dropdown`）。
  - 固定按鈕：`save_now`、`goto_report`、`new_record`。

### F-003 新增「氣味」檢查項
- **MVP 目標**：加入實際租屋中常見卻易忽略的嗅覺問題。
- **細節草案**：
  - 新增項目：
    - 是否有菸味（公共區域/室內）
    - 是否有煮菜油煙味（鄰戶或管道）
  - 加入提示文案（建議在哪些時段觀察更準）。
  - 納入報告與精簡版統計。
- **本輪定稿**：
  - 所屬分類：`環境鄰居`（`env_neighbor`）。
  - 輸入方式：三級（無異味 / 可接受 / 明顯異味）。
  - 顯示策略：保留「建議觀察時段」短提示（`need_hint`）。

### F-004 細節改為選單式輸入
- **MVP 目標**：降低文字輸入負擔，提升填寫率。
- **細節草案**：
  - 常用狀態改成 chips：
    - 良好 / 普通 / 偏差
    - 有 / 無 / 不確定
  - 保留「其他備註」自由輸入欄。
  - 選單值同步進報告細節。
- **本輪定稿**：
  - 第一批覆蓋：安全、設備、費用、環境（全高頻分類）。
  - 互動元件：單選 chips（`single_select_chip`）。
  - 備註策略：每個選單項都可補充備註（`keep_note_all`）。

### F-005 報告頁 UX 優化
- **MVP 目標**：手機上更快得到「是否值得租」判斷。
- **細節草案**：
  - 報告開頭固定顯示總評、分數、風險等級。
  - 詳細區塊預設收合，可展開看細項。
  - 比較與雷達區塊改為次要資訊，避免干擾第一判讀。
- **本輪定稿**：
  - 比較房源：獨立比較頁（`separate_page`）。
  - 細項預設：收合（`collapsed`）。
  - 風險訊號：強提示（色塊 + icon + 標籤，`strong`）。

### F-006 精簡版（文件版）優化
- **MVP 目標**：精簡版可直接交付，且匯出結果一致。
- **細節草案**：
  - 桌機版加強字級層次與欄寬比例。
  - 手機版預覽維持單欄，不跑版。
  - 強化評分標準說明（A/B/C 定義固定顯示）。
- **本輪定稿**：
  - 文件欄位：先不加簽名區（`no_signature`）。
  - 版面密度：平衡型（`balanced`）。
  - 手機預覽：盡量接近列印版（`near_print_match`）。

#### 0) 設計前置（使用者研究與規劃，必交付）
- **目標**：設計前釐清需求與結構，使查核表頁符合使用者期望，再動程式。
- **建議產出**（可放在 `project-docs/` 或設計工具連結）：

  2. **站點地圖（IA）**：僅兩個頂層 `查核表`／`報告` 下的區塊層級（摘要卡、紀錄管理、基本資料、分類清單、地圖、底部列等）與其從屬關係。
  3. **線框圖（低保真）**：手機首屏、捲動中段、展開單一查核項三種狀態；標註「權威資訊區」與「摘要／捷徑區」。
- **完成定義**：產出經過一輪走查（自己或 1～2 位使用者）後，列出「要合併的區塊對照表」再進開發切片。

#### 1) 明確的導覽流程（3 次點擊原則）
以下為 **查核表頁** 建議驗收的「主要任務」，每一項從進入查核表頁起算，**≤ 3 次主要互動**（點擊、開啟選單並選一項、開啟面板並關閉算一次流程）需可達成：
| 任務 | 範例終點 | 備註 |
|---|---|---|
| 確認／切換目前看房紀錄 | 紀錄已切換、標題可辨識 | 「找紀錄」等同全站搜尋的替代指標 |
| 編輯地址或月租金 | 焦點在對應欄位 | 不重複多個「同一用途」入口 |
| 開啟地圖選址 | 地圖面板可操作 | 入口唯一或主次分明 |
| 切換分類並開始勾選 | 某分類第一項可勾 | 與摘要區不重複搶同一資訊 |
| 前往報告 | 報告頁已載入同紀錄 | 與底部列、頂部切換一致 |

若任務無法滿足，優先調整 **資訊架構**（合併區塊、收合次要），再調視覺。

#### 2) 直觀的互動設計（慣例對應，勿重新發明標準）
本產品為 **工具型雙頁 App**，與大型內容站不同；慣例對應如下（實作時以此檢核，而非硬套不存在的功能）：

| 常見慣例 | 在 Rental Buddy 的對應 | 原則 |
|---|---|---|
| 左上角 Logo／品牌回首頁 | 頂欄左側產品識別：**可回到查核表「穩定起點」**（例如捲動頂部、關閉已開啟的選單／面板） | 不偷偷改行為；若僅捲動頂部需有視覺回饋 |
| 右上角搜尋 | **本階段若無全域搜尋**：以「紀錄切換／目前紀錄」為右上資訊尋找核心；未來若有搜尋再佔用同一認知位置 | 不放裝飾性假搜尋框 |
| 主要內容區 | 查核清單為主軸；摘要為輔助 | 主從分明 |
| 破壞性／少用動作 | 收在選單內或次要按鈕 | 降低誤觸 |

#### 3) MVP 目標（實作階段）
- **整合重複項目**：盤點查核表頁上「同一資料出現兩次以上」或「同一動作有兩個入口」者，合併為單一權威區或「摘要 + 展開」。
- **調整排版 UI/UX**：依線框調整間距、層級、閱讀動線；不擴充新功能範圍除非 IA 走查後明列。
- **與 F-002 關係**：F-002 偏版面元件；F-013 偏資訊與導覽邏輯，可回頭修正 F-002 時未處理的重複。

#### 4) 本輪定稿（草案，實作前可修）
- **設計門檻**：無線框不定大改（`wireframe_first`）。
- **重複處理策略**：權威單一來源 + 可選摘要（`single_source_optional_summary`）。
- **驗收**：上表五項任務皆通過 3 次互動檢核（`three_tap_audit`）。

#### 5) 驗收標準（Acceptance Criteria）摘要
- [ ] AC-1：完成「重複區塊對照表」並在查核表頁消弭表列重複（或改為摘要連結）。
- [ ] AC-2：五項核心任務皆滿足 3 次主要互動內可完成（手機實測）。

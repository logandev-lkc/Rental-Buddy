# 查核表 UX 優化（第三版）— 討論稿與規格

最後更新：2026-05-10（路徑 `v3/specs/checklist-ux-v3-spec.md`；總覽見 [README.md](../README.md)）

## 1) 文件目的

收錄第三版以**使用者體驗**為主的查核表調整方向：現場快速摘要、分類切換、各項目快速狀態與細節選項。本文檔為**討論稿**：問題描述與選項先記錄，定案後將對應條目移入 [decisions.md](../decisions.md)，並在 [feature-roadmap.md](../feature-roadmap.md) 更新狀態。

## 2) 與其他規格的關係

- **不改 v2 報告 JSON 大結構的前提**：除非為支援新互動而必須擴充欄位，否則優先維持現有 `reportData`／AI 匯入相容。
- **題目文案**：各項目文字可能持續微調；第三版著重在**版型、互動、規則一致性**，與文案並行時以本檔與決策為準。

## 3) 主題 A — 現場快速摘要（頂部區塊）

### 3.1 問題與意圖

| # | 問題／觀察 | 意圖 |
| --- | --- | --- |
| A1 | 地址常過長，現排版不易一眼看完 | 重新排版（換行、字級、標籤「地址」與內文分離），必要時摺疊次要片段 |
| A2 | 戶型類型與「房／廳／衛」三欄資訊重疊，又多為輸入欄 | 減少重複認知負擔；可能是「主類型 + 數字步進器」或單一結構化控件 |
| A3 | 廚房型態、室內坪數等同為輸入欄，區塊碎 | 將「結構化選項」與「自由數字／文字」分組，或同一張卡片內用統一元件族 |
| A4 | 戶型備註希望與主表格分流 | 備註獨立區（例如「看完上方條件後再填」），避免與必填項混在一起 |
| A5 | 房源條件 chips 項目多，且與下方分類查核有重疊 | 整理來源單一真相（SSOT）、或合併進分類、或縮成摘要列 + 詳情連結 |

### 3.2 待決方案（草案）

- **地址**：已於前端將地址改為**可多行文字區**（`textarea`），摘要區網格內地址欄**跨兩欄全寬**，支援換行與長字斷行；必要時後續再加「展開全文」。
- **戶型**：擇一主軸——「類型 dropdown + 房廳衛數字」或「房廳衛為主、類型為衍生顯示」，避免雙重輸入同一概念。
- **備註**：獨立 `Card` 或次區塊，文案引導「可於核對完戶型後補充」。

### 3.3 開放問題

- 房源條件與查核連動已部分存在（v2 Phase 3）：v3 是**只整理 UI** 還是**允許減少 chip 數量**（需對照報告權重）？

## 4) 主題 B — 分類切換

### 4.1 問題與意圖

| # | 問題／觀察 | 意圖 |
| --- | --- | --- |
| B1 | 「全部」像雞肋：沒選就是全部 | 預設不選＝顯示全部；選一個以上＝僅顯示所選分類（多選模式） |
| B2 | 目前除「全部」外有五個分類；第六個槽是否改為「篩選條件」 | 需評估：篩選維度（狀態／風險／未完成）與「分類」不同，不宜混在同一 row 無標示 |

### 4.2 待決方案（草案）

- **移除「全部」為獨立選項**：預設視為全選；使用者點分類 chips 時進入「篩選中的分類」狀態（需視覺區分：例如「已選 2 個分類」）。
- **第六槽**：方案甲保留給第六個**內容分類**（若產品擴充）；方案乙改為「篩選」入口（僅顯示未完成、僅顯示已標記需注意等）——需另開 **D-00x**。

### 4.3 開放問題

- 多選分類時，空狀態是否允許（0 個選中）？建議：**不允許 0**，取消最後一個選中時回到「等同全部」。

## 5) 主題 C — 快速狀態與細節選項

### 5.1 快速狀態：三級是否足夠

- 現況認知（扣掉「尚未確認」）：**良好／可接受／需注意**（類似 +3／+1／-1 的情緒錨點）。
- **討論點**：三級是否太少？若加第四級，會增加掃描成本與報告對應難度。
- **建議先定義語意**（寫入決策前可先共識）：
  - **良好**：無明顯扣分點、細節選項多為正向或中性。
  - **可接受**：可租但有妥協點，或資訊不足但無紅旗。
  - **需注意**：已勾選風險／扣分向細節，或手動標為需注意。

### 5.2 單題互動：點擊區塊／旗幟／問號（已稽核程式）

稽核來源：`rental-buddy-web/src/app/app.ts`（`toggleCheck`／`toggleFlag`／`toggleExpand`）、`app.html`（查核項 `<article>`）。

| 操作 | 實際行為（2026-05-10） |
| --- | --- |
| 點擊題列主區（`(click)="toggleCheck"`） | **切換**「已確認 ↔ 未確認」：`checked` 反轉；勾選時 `quickStatus='good'`、`flagged=false`；取消勾選時 `quickStatus='unknown'`。（**不是**單向「永遠設為良好」。） |
| 點擊 ⚑ 旗幟（`toggleFlag`） | **切換**需注意：`flagged` 反轉；若為需注意則 `checked=false`、`quickStatus='attention'`；取消旗幟則 `quickStatus='unknown'`。 |
| 點擊 ?（`toggleExpand`） | **切換**展開／收合：展開後才顯示提示文案、**快速狀態四鍵**、**細節選項**、備註輸入（未展開時僅見標題與左側勾選框）。 |

補充：左側小格顯示 ✓（checked）或 !（flagged），與主區點擊連動同一組狀態。

### 5.2.1 細節選項變更後的 quickStatus（`syncQuickStatusFromOptions`）

具細節 chips 的題目（`itemOptionConfig` 非空）在變更選項時會呼叫 `syncQuickStatusFromOptions`（`app.ts`）：

| 條件 | 結果 |
| --- | --- |
| 未選任何細節選項 | `quickStatus='unknown'`，未勾選、未標旗 |
| 任一選項字串命中 `riskOptionKeywords`（含「需注意」「不足」「老舊」…） | `quickStatus='attention'`，`flagged=true` |
| 已選至少一項且**無**風險關鍵字 | `quickStatus='ok'`（UI 顯示「**可接受**」）、`checked=true`、`flagged=false` |

因此：**僅選正向細節、無風險字樣時，系統設計為「可接受」而非「良好」。**「良好」主要來自題列勾選（或手動點快速狀態「良好」）。是否改版為「全正向 → 良好」屬產品決策（候選 D-003），與 §7 矩陣一併討論。

### 5.3 細節選項：單選／多選與顏色語意

- **單選 vs 多選**：應由題目型別決定（結構化資料已定義者）；若同一題混合「正向堆疊」與「風險勾選」，演算法需定義**優先序**（現行程式：**任一字串命中風險關鍵字 → 需注意**，見 §5.2.1）。
- **底色**：建議分離兩種語意，避免使用者混淆：
  - **風險／扣分向**：紅系（底 + 字對比足夠）。
  - **提示／待確認／中性**：琥珀或藍灰系，**避免與風險紅並用同一套「紅底」**。
- **範例（廚房設備 f6）**：僅選「有冰箱、有爐具、抽風佳」時出現「可接受」——與 §5.2.1 規則一致；若期望為「良好」需另立規則或調整聚合邏輯。

### 5.4 何時先寫進文件、何時邊做邊定

| 議題 | 建議 |
| --- | --- |
| 三級定義、風險色 vs 中性色 | **實作前**至少有一頁共識（本檔 §5 + decisions），否則報告與評分會翻來覆去 |
| 點擊／旗幟／展開是否改版 | **現況稽核已完成**（§5.2）；若要改版互動再開原型驗證 |
| 聚合規則（為何全正向仍「可接受」） | **實作前**補矩陣；可從最容易引發誤解的 5～10 題開始 |

## 6) 建議驗收方向（草案）

- **摘要區**：在主流手機寬度下，地址與戶型資訊無需橫向捲動即可閱讀主要內容。
- **分類**：預設等同「全部」；多選時列表僅出現所選分類之題目；狀態列清楚。
- **狀態與細節**：任一題目的快速狀態與細節選項組合，可依公開規則預期結果（見 §7，持續補題）。

## 二、評等（四級 + 未評）

使用者面向標籤與長提示（`quickStatusOptions`，選項上的 `title` 亦供長按／hover）：

| value | 標籤 | 長提示（摘要） |
|-------|------|----------------|
| `good` | 良好 | 大致滿意、未見明顯扣分點（+2）。已選時再點「良好」可回到未評。 |
| `ok` | 可接受 | 大致可行，有可談的小妥協或資訊落差（+1）。已選時再點「可接受」可回到未評。 |
| `attention` | 需注意 | 有疑慮或待核對，建議追問或列待辦（-1）。已選時再點「需注意」可回到未評。 |
| `bad` | 很不理想 | 狀況偏差或有明顯硬傷，建議慎重評估（-2）。已選時再點「很不理想」可回到未評。 |
| （無按鈕） | 尚未確認 | `quickStatus === 'unknown'`；未選任一評等即為未評。 |

**系統衍生（儲存／報表）**：`good`／`ok` 對應 `checked`；`attention`／`bad` 對應 `flagged`（與舊「問題／風險」語意相容）。

## 四、細節選項：結構與行為

- **有 `itemOptionConfig[id]`**：題型為 **multiSelect**（可多選細節 chip）。更動選項會呼叫 `syncQuickStatusFromOptions()`（見下節 **程式現狀**）。
- **無設定**：題型為 **singleSelect**，僅靠四級評等，無細節 chip。
- **風險 chip**：選項文字若包含下列關鍵片段之一，UI 以「風險」樣式標示（`isRiskOption`）。關鍵字列表見 **§五**。
- **與房源條件連動**：題目 id **`c1`、`c3`、`c4`、`n4`** 與房源條件欄位雙向／單向同步（押金、管理費、租補、管理員等）。調整題目文案或選項時需一併檢查 `syncPropertyFieldsFromLinkedChecklist` 相關邏輯（見程式）。

### 4.1 程式自動同步評等（現狀，`syncQuickStatusFromOptions`）

若該題有細節選項，系統**只會**依規則把評等設成下列三者之一（**不會**自動設成「良好」或「很不理想」）：

| 條件 | 自動設定的評等 | `checked`／`flagged` |
|------|------------------|----------------------|
| 未選任何細節選項 | `unknown`（尚未確認） | 皆 false |
| 已選選項中，**至少一個**含 `riskOptionKeywords` 子字串 | `attention`（需注意） | flagged |
| 已選選項皆**不含**風險關鍵字 | `ok`（可接受） | checked |

因此：**只要沒命中風險關鍵字，不論語意上較好或較差，程式目前一律同步為「可接受」。** 使用者仍可手動改評等 chip（良好／需注意／很不理想等）；但若再次觸發同步（例如增刪細節選項），評等會再被規則覆寫回上表。

### 4.2 現狀與直覺差異（審稿時請特別注意）

以下為常見「語意上應分級」但 **程式未自動對應「良好／很不理想」** 的例子（應靠使用者手動評等，或未來改程式對應表）：

- **f1 冷氣是否為變頻款**：`無冷氣` 不含風險關鍵字 → 現狀仍會被同步成 **可接受**，與直覺（通常偏負面）不一致。
- **f1**：`變頻冷氣` 亦會同步成 **可接受**，與直覺（通常為較佳）不一致。
- **f2 熱水**：`忽冷忽熱` 若未命中關鍵字 → 現狀可能同步為 **可接受**（直覺上常應為需注意）。
- **多選題**：若同時選「正向」與「風險」選項，任一風險命中 → 整題會被同步為 **需注意**（符合「 worst wins」直覺）。

### 4.3 審稿建議：細節選項 ↔ 四級評等語意對照（**完整：`itemOptionConfig` 內每一題**）

供 ChatGPT／Claude 檢查文案與產品邏輯是否一致。

- **建議語意（審稿）**：依租屋情境推敲的「良好／可接受／需注意／很不理想」參考，可再調整。
- **程式現狀（自動同步）**：依 `riskOptionKeywords` 子字串匹配；命中任一風險片段→同步為「需注意」，否則（有選項時）→「可接受」。**程式不會自動同步「良好」或「很不理想」。**
- **複選題**：同一題可複選時，建議以「語意較差／風險較高者」理解整題；程式端只要選取集合中**任一**選項命中風險字→整題「需注意」。
- **重要程式細節**：關鍵字 `異味` 會命中 **`無異味`**（因字串包含「異味」二字），導致程式將該選項視為風險→**需注意**。此為字串匹配副作用，審稿時應列為已知問題或後續改規則。

### 4.4 給審稿者的結論句

- **現狀**：細節選項 → 評等，本質是「**二分的風險字串匹配 + 預設可接受**」，不是四級完整對照表。
- **若要對齊直覺**（例如變頻→良好、無冷氣→很不理想）：需另開需求——調整 `riskOptionKeywords`、為特定題目加對照表，或允許選項字串映射到 `good`／`bad`。

## 八、其他系統關聯（審稿時一併考量）

- **雷達五軸**：`contract`／`facility`／`safety`／`living`／`neighbor` 與 `categoryScoreWeightByAxis` 加權。
- **查核項得分係數**：與 `quickStatus`、legacy `checked`／`flagged` 等有關（見 `getItemQuickScoreFactor`）。
- **報表匯出／AI JSON**：題目結構含 `answerType`、`answers`／`selectedOptions`、`quickStatus`（程式註解與序列化處請搜 `ChecklistAnswerType`、`HouseRecord.state`）。

## 九、給審稿 LLM 的建議提示

1. 維持 **同一題** 上「標題／tip／細節選項」語氣一致；細節選項避免與 `riskOptionKeywords` 意外誤判（除非本意為風險）。
2. 修改 **c1、c3、c4、n4** 選項文字時，務必對照房源條件欄位與同步函式，避免連動失效。
3. 新增題目時：需新增 `items` 條目，並視需要新增 `itemOptionConfig`、`itemRiskConfig`、`itemReportCopyConfig`、`itemReportPriorityBoost`。
4. 審閱「細節選項」與「良好／可接受／需注意／很不理想」是否一致時，請先讀 **§4.1～4.3**：區分 **程式現狀**與 **建議語意**；若要產品與直覺完全一致，需另行規格化（對照表或程式變更），不是改文案 alone 就能達成。

## 十、AI 審稿回覆格式（供貼給工程／Coding Agent 實作）

審稿 LLM 請**優先輸出「單一 JSON 程式碼區塊」**（語言標籤用 `json`），內容可被人工或自動化對照到 `app.ts`／`app.html` 的實際欄位。若無任何可實作變更，仍輸出 JSON，`changes` 可為 `[]`，並在 `summary` 說明「無建議變更」。

### 10.1 頂層結構（必填鍵名）

| 鍵 | 型別 | 說明 |
|----|------|------|
| `schemaVersion` | `number` | 固定為 `1`（日後改版遞增）。 |
| `summary` | `string` | 審稿結論一句話（給人看）。 |
| `changes` | `array` | 見 **§10.2**；每筆代表一項可實作的文案或設定變更。 |
| `riskKeywordReview` | `array`（選填） | 見 **§10.4**；建議增刪 `riskOptionKeywords` 子字串時使用。 |
| `blocked` | `array`（選填） | 需改程式邏輯／規格、**無法**只靠改字串完成的項目（每筆含 `reason` 與可選 `relatedItemIds`）。 |
| `openQuestions` | `array`（選填） | 需產品／工程決策的問句字串。 |

### 10.2 `changes[]` 每一筆的共通欄位

| 鍵 | 必填 | 說明 |
|----|------|------|
| `id` | 是 | 字串，建議 `chg-001`、`chg-002`… 方便討論引用。 |
| `scope` | 是 | 見下表 **允許值**；決定要改哪一段程式對應的資料。 |
| `rationale` | 是 | 為何要改（審稿理由，一句以上）。 |
| `implementationHint` | 否 | 給實作者的提醒（例如：「改 n5 選項時檢查 `異味` 子字串誤判」）。 |

**`scope` 允許值與實作對照**

| `scope` | 實作位置（約定） | 該筆建議一併提供的鍵 |
|---------|------------------|----------------------|
| `checklistItem` | `app.ts` → `items` 中 `id` 相符的物件 | `itemId`，`field`：`title` \| `tip`，`proposed`（新全文） |
| `detailOptions` | `app.ts` → `itemOptionConfig[itemId]` | `itemId`，`proposedOptions`（**完整**字串陣列，順序即 UI 順序） |
| `quickStatusLabel` | `app.ts` → `quickStatusOptions[].label` | `quickStatus`：`good` \| `ok` \| `attention` \| `bad`，`proposed` |
| `quickStatusTitle` | `app.ts` → `quickStatusOptions[].title` | 同上，`proposed` |
| `checklistFilterLabel` | `app.ts` → `checklistFilterOptions` | `filterId`：`pending` \| `confirmed` \| `good` \| `ok` \| `attention` \| `bad` \| `priority_high` \| `priority_normal` \| `priority_later`，`proposed` |
| `checklistFilterGroupLabel` | `app.ts` → `checklistFilterGroups` | `groupId`：`status` \| `quick` \| `priority`，`proposed` |
| `categoryLabel` | `app.ts` → `categories[].label` 或 `categoryMap` | `categoryId`：`contract` \| `facility` \| `safety` \| `living` \| `neighbor`，`proposed` |
| `itemReportCopy` | `app.ts` → `itemReportCopyConfig[itemId]` | `itemId`，`field`：`positiveText` \| `riskText` \| `nextAction`，`proposed` |
| `itemRisk` | `app.ts` → `itemRiskConfig[itemId]` | `itemId`，`weight`（數字），`riskLevel`：`low` \| `medium` \| `high` |
| `itemReportPriorityBoost` | `app.ts` → `itemReportPriorityBoost[itemId]`（若該鍵存在） | `itemId`，`boost`（數字） |
| `notePlaceholder` | `app.html` → 查核卡備註 `textarea` 的 `placeholder` | `proposed`（單一字串） |
| `propertyField` | `app.ts` → `propertyChoiceGroups` 內某一 `field` 的 `label` 或 `options[]` | `propertyField`（如 `depositMonths`），若改選項則 `proposedOptions`（完整陣列）；若只改群組標題則 `proposed` |

**相容性（選填，但建議填）**

- `previous`：實作前用於核對的舊字串或舊陣列（與 repo 一致時可降低改錯檔風險）。

### 10.4 `riskKeywordReview[]`（選填）

當審稿認為應**新增／移除** `riskOptionKeywords` 中的子字串（以影響 `syncQuickStatusFromOptions`）時，每筆使用：

| 鍵 | 說明 |
|----|------|
| `action` | `add` \| `remove` |
| `keyword` | 子字串（與程式 `option.includes(keyword)` 語意一致）。 |
| `rationale` | 為何要加／刪。 |
| `affectedItemIds` | 選填：預期主要影響的題目 id（如 `n5`）。 |

實作時仍須人工檢查是否造成新的誤判（參考 §4.3 `無異味`／`異味` 案例）。

### 10.5 簡易備援格式（無法產 JSON 時）

若環境不便輸出 JSON，可用 **Markdown 表格**，欄位固定為：`id` | `scope` | `鍵值欄位…` | `proposed` | `rationale`；`scope` 仍使用 **§10.2** 的英文代碼。實作者再手動轉成程式變更。

### 10.6 實作者檢查清單（收到 AI 輸出後）

1. **§4.1～4.3**：凡動到 `detailOptions` 文案，用 `riskOptionKeywords` 逐字檢查是否意外命中或漏判。
2. **連動題**：修改 **c1、c3、c4、n4** 的選項文字時，對照 `syncPropertyFieldsFromLinkedChecklist`／`reconcileLinkedChecklistFromProperty` 等邏輯是否仍可比對。
3. **報表／PDF**：動到 `itemReportCopy` 或 `itemRisk` 時，快速走查報告頁與匯出段落。
4. **型別**：`quickStatus`、`riskLevel`、`filterId`、`categoryId` 必須為上表列舉值，大小寫一致。
